# REST란?
- REST는 REpresentational State Transfer의 약자
- 웹과 같은 분산 하이퍼미디어 시스템의 장점을 활용할 수 있는 아키텍처


- 자원(Resource) - URI
- 행위(Verb) - HTTP Method
- 표현(Representations)

# REST의 특징
1. Uniform interface
    - URI로 지정한 리소스에 대한 조작을 통일되고 한정적인 인터페이스로 수행
2. Stateless
    - 작업을 위한 상태정보를 따로 저장하고 관리하지 않음.
3. Cacheable
4. Client-Server구조
    - 클라이언트와 서버에서 개발할 내용을 명확히해 서로간 의존성을 줄인다.
5. 계층형 구조
    - 다중 계층으로 구성될 수 있으며 보안, 로드밸런싱, 암호화 계층을 추가해 구조상의 유연성을 둘 수 있고 Proxy, 게이트웨이 같은 네트워크 기반의 중간 매체를 사용할 수 있다.
6. code-on-demand (Optional)
    - 서버에서 코드를 클라이언트로 보내서 실행할 수 있어야 한다. (js)

http만 잘 따라도 대부분은 잘 지킨다.
# 잘 지켜지지 않는 조건들
Uniform interface의 제약조건 
- self-descriptive
    - REST API 메시지만으로 API가 증명되어야 한다는 것입니다.
- HATEOAS(Hypermedia As The Engine Of Application State)
    - 애플리케이션의 상태는 Hyperlink를 이용해 전이되어야 한다

### 온전한 REST API를 구현하기 위해서

JSON을 반환하는 API를 구현했는데 위 조건들을 지키지 못했다.
#### 개선해야 할 점.
- self-descriptive
  - host를 나타내는 헤더 추가
  - IANA에 등록 또는 JSON 명세에 대한 Link 헤더를 추가해야 한다
- HATEOAS
  - JSON 데이터와 헤더에 하이퍼링크를 추가해 애플리케이션의 상태를 표현해주어야 한다

# HTTP 응답 상태 코드
- 1xx (조건부 응답) : 요청을 받았으며 프로세스를 계속한다
- 2xx (성공) : 요청을 성공적으로 받았으며 인식했고 수용했다
  - 200 : 클라이언트의 요청을 정상적으로 수행
  - 201 : 클라이언트가 어떠한 리소스 생성을 요청하고 성공적으로 생성됨 (POST)
- 3xx (리다이렉션 오류) : 요청 완료를 위해 추가적인 작업이 필요하다
  - 301 : 클라이언트가 요청한 리소스에 대한 URI가 변경돠었을때 사용하는 응답코드
- 4xx (클라이언트 오류) : 요청의 문법이 잘못되었거나 요청을 처리할 수 없다
  - 400 : 클라이언트의 요청이 부적절할 경우
  - 401 : 클라이언트가 인증되지 않은 상태에서 보호된 리소스를 요청했을때
  - 403 : 유저 인증상태와 관계없이 응답하고 싶지 않은 리소스를 클라이언트가 요청했을때
  - 405 : 클라이언트가 요청한 리소스에서는 사용 불가능한 method를 이용했을경우
- 5xx (서버오류) : 서버가 명백히 유효한 요청에 대해 충족을 실패했다.
  - 500 : 서버에 문제가 있을 경우
