# 디자인패턴은 왜 필요할까?

- 추후 재사용, 호환, 유지보수시 발생하는 문제를 검증된 방법으로 효율적으로 코드를 개선할 수 있고,
- 패턴을 정의함으로써 개발자들이 협업할때 **효율적으로 의사소통** 할 수 있는 장점도 있다~

# 싱글톤 패턴
애플리케이션이 시작될 떄, 어떤 클래스가 최초 한 번만 메모리를 할당하고 해당 메모리에 인스턴스를 만들어 사용하는 패턴
- 즉, 하나의 인스턴스만 생성하여 사용하는 패턴이다. (계속 생성자를 호출해도 이미 생성된 단일 객체를 반환)
- 싱글톤을 구현한 인스턴스는 전역이므로, 다른 클래스의 인스턴스들이 데이터를 공유하는것이 가능하다는 장점이 있다.

### 언제 사용하나요?
주로 공통된 객체를 여러개 생성해서 사용해야하는 상황

### 주의해야하는 문제점
싱글톤 인스턴스가 너무 많은 일을 하거나 많은 데이터를 공유시킬 경우
- 다른 클래스들 간의 결합도가 높아져 "개방-폐쇄원칙"을 위배하게 된다.
- 결합도가 높아지면 유지보수와 테스트가 어려워진다.

멀티스레드 상황에서 동기화 처리를 하지 않았을 때
- 인스턴스가 2개 생성되는 문제가 발생할 수 있다.
- 해결하기 위해 os에서는 mutual exclusion 적

# 어댑터 패턴
클래스를 바로 사용할 수 없는 경우가 있다. 이때 중간에서 변환 역할을 해주는 클래스가 필요하다.

상속을 통해
호환되지 않은 인터페이스를 사용하는 클라이언트에 그대로 활용이 가능하다
향후 인터페이스가 바뀌더라도 변경 내역은 어댑터에 캡슐화되므로 클라이언트는 바뀔 필요가 없다.

# 템플릿 메소드 패턴
로직을 단계별로 나눠야 하는 상황에서 적용한다
단계별로 나눈 로직들이 앞으로 수정될 가능성이 있을 경우 더 효율적이다.
조건
- 클래스는 추상(abstract)로 만든다
- 단계를 진행하는 메소드는 수정이 불가능하도록 final키워드를 추가한다.
- 각 단계들은 외부는 막고, 자식들만 활용할 수 있도록 protected로 선언한다

# 옵저버 패턴
상태를 가지고 있는 주체 객체와 상태의 변경을 알아야하는 관찰 객체
한 객체의 상태가 바뀌면 그 객체에 의존하는 다른 객체들에게 연락이 가고 자동으로 정보가 갱신되는
1:N 관계를 정의한다.
인터페이스를 통해 연결하여 느슨한 결합성을 유지하며, Publisher와 Observer 인터페이스를 적용한다.
Publisher에 Subscriber가 구독하면 출판사가 업데이트 된 내용을 모든 구독자들에게 등록시켜준다.

# 스트레티지 패턴
어떤 동작을 하는 로직을 정의하고, 이것들을 하나로 묶어 관리하는 패턴 // 캡슐화
새로운 로직을 추가하거나 변경할 때, 한번에 효율적으로 변경이 가능하다.
수업에서는 메서드를 클래스화 하는것이라고 배웠다.
