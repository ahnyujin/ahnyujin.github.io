Thread가 나온 이유

1.Process는 무겁다.

- 프로세서가 동작하기 위해서 많은 리소스를 가지고 있다. ex) address space

    프로세스는 실행중에 있는 프로그램이며 프로그램이 메인 메모리에 올라와서 동작을 해야한다. 그러한 동작을 가상 메모리 환경에서 진행한다. 가상 환경을 운영 하기 위해서는 page table이 필요하다.( page table로 운영되는 주소 공간을 가져야 한다.) 프로그램이 동작하면서 열었던 운영체제의 자원들 프로세스의 계정 정보들. 이러한 정보들은 pcb에 의해서 관리된다.

- 프로세스가 처음 만들어질 때 메모리를 할당받는다. 그리고 초기화를 진행.
- IPC ( Inter Process communication ) 비용이 크다

IPC

1. shared memory 방식

2. message passing방식

프로세스의 전체 흐름을 스레드라고 한다. 하나의 프로세스는 기본적으로 하나의 스레드를 가지고 있다.

단일 프로세스로 웹 솔루션을 구현한다면

브라우저나 사용자의 컴퓨터가 인터넷을 통해 웹 요청을 하면 이 프로세서가 요청을 해석해서 내부의 데이터베이스를 검색해서 필요한 정보를 다시 줄 것이다. HTTP/HTTPS 프로토콜로. 웹 페이지 요청 리퀘스트 자체가 시리얼라이즈 된다. 하나의 프로세스밖에 없기 때문에 하나의 리퀘스트를 처리할 때 다른 요청을 처리 할 수 없다. 따라서 얘를 큐에 쭉 저장했다가 하나씩 처리를 하게 된다. 이 경우에 response time이 많이 느려질 수 있다.

그래서 이러한 문제를 해결하기 위한 방법으로 하나의 리퀘스트에 대해 하나의 프로세스를 할당하는 방식. 즉 각 request에 대해 프로세스를 새로 생성하는 방식.

```c
while(1) {
	int sock = accept(); // request가 도착하기를 기다림
	if ((pid = fork()) == 0) { //fork로 프로세스 생성, child process일때 동작 처리
		// Handle client request
	} else {
		// Close socket
	}
}
```
