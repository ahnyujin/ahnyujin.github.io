다른 객체지향언어인 C++에서는 다중상속(multiple-inheritance)이 가능한데 자바에서는 왜 단일상속만 가능할까?

# issue1 : 다중상속에는 [다이아몬드 문제](http://en.wikipedia.org/wiki/Diamond_problem) 가 있다.
> [다이아몬드 문제]
>
> 두 클래스 B와 C가 A에게 상속받고, 클래스 D가 B,C에게 상속받을 때 발생하는 모호함.
>
> : A의 메서드를 B, C가 각각 오버라이드하고, D는 오버라이드 하지 않는다면
>
>  D는  B와 C중 누구의 메서드를 상속받을까?

## 언어별 해결 방법
### C++의 경우
C++는 기본적으로 각각의 상속 path를 분리한다. D 객체는 실제로 두 개의 개별 A 객체가 포함되며 A의 멤버를 적절히 사용해야한다.

A를 **virtual** 로 상속받으면
```cpp
class A {
 public:
  // ...
};
class B : public virtual A {
  // ...
};
class C : public virtual A {
  // ...
};
class D : public B, public C {
  // ...
};
```
D에서 다중 상속 시에도, 컴파일러가 언제나 A를 한 번만 포함하도록 지정할 수 있게 된다.

가상 상속과 비가상 상속을 혼합할 경우 단일 가상 A와 A에 대한 각 비가상 상속 경로에 대한 비가상 A가 생성된다. C++에서는 사용할 기능이 어떤 부모 클래스에서 호출되는지 명시적으로 가리켜야한다.
가상 상속 메커니즘을 통해 여러 클래스의 단일 인스턴스를 만들 수 있다.
### Java 8
인터페이스에 디폴트 메소드 사용. A, B, C가 인터페이스인 경우 B, C는 각각 A의 추상메서드에 다른 구현을 해서 다이아몬드 문제를 불러일으킬 수 있다. 따라서 클래스 D는 메서드를 오버라이드 해야한다.
( Java 8 이전에는 인터페이스에 디폴트 메서드가 없었기 때문에 이런 문제가 없었다. )

# issue2 :  상속대신 컴포지션을 사용하자
- 이펙티브 자바 18,19 참고

상속은 코드를 재사용할 수 있는 좋은 아이디어지만, 템플릿 메소드 패턴처럼 상속해서 쓰도록 고려된 객체가 아니라면 부작용이 있을 수 있다. 캡슐화가 깨져 조상 객체와 피상속 객체 사이에 강한 결합이 생길 가능성이 있는 것이다. 이런 부작용을 막기 위해 컴포지션 클래스의 public 메서드를 final로 만들어 오버라이드하지 못하게 막기도 한다. 아니면 최소한 상속했을 때 문제가 생기지 않도록 내부 정보를 문서화하라고도 한다.

# 결론 : interface 다중 구현 허용
상속이 이렇게 위험하니 다중상속은 얼마나 더 위험할까?

하지만 다중상속은 매우 유용하다. **단일상속으로는 객체의 복합적 특성을 고려할 수 없기 때문**

따라서 자바는 부모가 순수 추상 객체(interface)일때 다중 구현을 허용한다.

자바의 방식은 다이아몬드 문제뿐 아니라 다중 상속의 복잡성도 해결할 수 있다.

단일상속을 사용함으로써 클래스간의 관계를 명확히 하고 코드를 신뢰할 수 있게 만든다.
